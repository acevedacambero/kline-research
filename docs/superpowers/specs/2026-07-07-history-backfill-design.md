# 沪深短历史缓存补全设计

## 目标

为沪深缓存中少于 250 个有效交易日的证券提供可恢复、可审计的历史补全任务。任务必须绕过只返回近期窗口的腾讯股票行情，使用新浪/AKShare 长历史源从 `1990-01-01` 拉取数据，并复用现有复权和快照管线。

## 范围

- 只处理上海、深圳市场；北交所继续排除。
- 只处理当前缓存少于 250 个有效交易日的证券。
- 完整历史仍少于 250 日且数据覆盖至近期的证券视为新股，记录为 `listing_history_short`，不在后续补全任务中反复下载。
- 补全完成后不自动生成 P1 标签或 P2 特征；用户检查任务结果后使用现有按钮生成。
- 首期不增加任意单票输入、日期范围编辑或自动定时调度。

## 架构与数据流

新增 `HistoryBackfillService`，负责扫描、筛选和补全，不把规则继续堆入 API 路由。

1. 从 DuckDB 清单读取沪深规范化 Parquet 路径。
2. 读取每个文件的有效日期数、最早日和最晚日；已记录 `listing_history_short` 且内容指纹未变化的证券跳过。
3. 将少于 250 日的证券形成不可变任务清单。
4. 对任务清单调用显式长历史接口；股票行情禁止回退到腾讯短窗口。
5. 复权因子仍使用现有新浪因子接口，随后调用 `DatasetPipeline.import_security` 原子生成新快照并更新清单。
6. 补全结果达到 250 日时记为 `completed`；仍不足 250 日但覆盖至最近 10 个自然日内时记为 `listing_history_short`；否则记为错误并保留旧快照。

旧快照只有在新数据和复权派生全部成功后才失去“最新”引用，因此中途失败不会破坏可用数据。

## API 与任务模型

新增接口：

- `POST /api/datasets/backfill-history`：启动任务，返回任务 ID、候选数和阈值。
- `GET /api/datasets/backfill-history/{task_id}`：返回 `status`、`done`、`total`、`completed`、`listingHistoryShort`、`errors[]`、`currentSecurity`、速度和 ETA。

补全任务与导入、P1、P2共享现有重任务协调边界。任一写任务运行时启动补全返回 `409`，补全运行时也阻止其他写任务，确保生产 DuckDB 只有一个写者。

错误必须包含证券键、阶段和稳定错误码；单票失败不终止整个任务。任务重启后重新扫描清单，已成功补全和已确认新股的证券会被跳过。

## 前端

数据状态面板新增“补全短历史”按钮。点击后展示：

- 候选证券数；
- 已完成/总数；
- 已补全数；
- 新股历史不足数；
- 错误数、当前证券、速度和 ETA。

任务结束后提示用户检查错误，再按需点击现有“生成 P1 标签”和“生成 P2 特征”。不自动串联计算任务。

## 数据质量与审计

质量事件新增：

- `history-backfilled`：记录补全前后有效日数量和数据源；
- `listing-history-short`：记录有效日数量、最早日、最晚日和判定版本；
- `history-backfill-failed`：记录阶段和脱敏错误信息。

规则版本固定为 `history-backfill-v1`。质量接口增加短缓存、新股历史不足和补全失败的汇总计数。

## 测试与验收

- 单元测试：候选筛选只包含沪深短缓存；正常缓存和已确认新股跳过。
- 数据源测试：补全股票行情只走长历史新浪/AKShare接口，不调用腾讯短历史。
- 管线测试：成功补全更新快照；下载、因子或派生失败时旧快照保持可用。
- 任务测试：单票失败隔离、进度统计、重启扫描和所有写任务互斥。
- API 测试：启动、轮询、无候选、冲突和错误结构。
- 前端测试：按钮、轮询、终态统计及后续 P1/P2 提示。
- 生产验收：先对少量上海短缓存执行；人工核对日期范围与 K 线后再处理全部候选。最终要求旧上海证券达到至少 250 日，新股带明确审计状态，失败证券可追踪。

## 安全边界

- 不删除旧快照，不直接修改 Parquet。
- 不扩大公网接口权限；接口继续受 Cloudflare Access 和后端 JWT 校验保护。
- 不记录 Access JWT、Tunnel token 或上游响应中的敏感内容。
- 不修改现有 Nginx、Tunnel 或共享数据目录权限。
